Demo 1: Iceberg Table — Create & Query

Create an Iceberg table, INSERT sample data, and run queries. The table looks and feels like a normal PostgreSQL table, but data is stored as Parquet files on S3 with Iceberg metadata.

default_location_prefix Data is written to: loading...
What Setup does:
  1. CREATE TABLE ... USING iceberg — creates an Iceberg table. Data lives on S3, not local disk.
  2. WITH (partition_by = 'day(event_date)') — pg_lake organizes Parquet files by day, enabling efficient date-range queries.
  3. INSERT 50,000 sample events spanning 60 days.
Check S3: After Setup, open AWS Console → S3 and navigate to your bucket. You'll see a directory structure like s3://bucket/.../demo/events/.../data/ with UUID-named folders containing Parquet files. This confirms PostgreSQL INSERT actually wrote data to S3.
Query: Daily event counts
SELECT date_trunc('day', event_date) AS day,
       count(*) AS events,
       count(DISTINCT region) AS regions,
       avg(amount)::numeric(10,2) AS avg_amount
FROM demo.events_hot
WHERE event_date >= now() - interval '7 days'
GROUP BY 1 ORDER BY 1;
Reading the EXPLAIN output: You'll see Custom Scan (Query Pushdown) / Engine: DuckDB / READ_PARQUET — this means pg_lake pushed the query down to DuckDB which reads Parquet files directly from S3. The Filters: line shows the date range was pushed down too, so only relevant partitions are scanned.
Single-row operations (INSERT, UPDATE, DELETE)
-- INSERT a single event
INSERT INTO demo.events_hot (event_date, region, category, amount, status)
VALUES (now(), 'Tokyo', 'manual_test', 999.99, 'active');

-- UPDATE it
UPDATE demo.events_hot SET amount = 1234.56
WHERE category = 'manual_test' AND region = 'Tokyo';

-- Verify
SELECT * FROM demo.events_hot WHERE category = 'manual_test';

-- DELETE it
DELETE FROM demo.events_hot WHERE category = 'manual_test';
Demo 2: Execution Plans — Date Range & DuckDB Pushdown

Compare execution plans and response times as you widen the date range. pg_lake's Iceberg tables push filters down to DuckDB, which reads only the relevant Parquet partitions from S3.

A: Last 1 day (narrow range, few partitions)
SELECT region, count(*), avg(amount)::numeric(10,2) AS avg_amount
FROM demo.events_hot
WHERE event_date >= now() - interval '1 day'
GROUP BY region ORDER BY count DESC;
B: Last 7 days
SELECT region, count(*), avg(amount)::numeric(10,2) AS avg_amount
FROM demo.events_hot
WHERE event_date >= now() - interval '7 days'
GROUP BY region ORDER BY count DESC;
C: Last 30 days
SELECT region, count(*), avg(amount)::numeric(10,2) AS avg_amount
FROM demo.events_hot
WHERE event_date >= now() - interval '30 days'
GROUP BY region ORDER BY count DESC;
D: Full table scan (no filter)
SELECT region, count(*), avg(amount)::numeric(10,2) AS avg_amount
FROM demo.events_hot
GROUP BY region ORDER BY count DESC;
What to observe:
  • All queries use DuckDB / READ_PARQUET — pg_lake pushes the entire query to DuckDB
  • The Filters: line changes with each date range — DuckDB only reads matching Parquet partitions
  • Estimated Cardinality increases as the range widens
  • Execution time grows with wider ranges (more Parquet files to read from S3)
pg_incremental & pg_cron — Continuous Sync Tools

Demo 3 uses two PostgreSQL extensions to continuously sync data from a heap table to an Iceberg table:

ExtensionRole
pg_incremental Tracks which time windows have been processed and runs an INSERT...SELECT command for each new window. Uses $1 (start) and $2 (end) parameters. Exactly-once semantics — progress and data are committed in the same transaction.
pg_cron Schedules pg_incremental pipelines to run automatically at intervals (e.g., every minute). Similar to Unix cron but runs inside PostgreSQL.
Note: pg_cron must be created in the postgres database. If unavailable, pipelines can be run manually with CALL incremental.execute_pipeline().
Pipeline types in pg_incremental:
  • Time interval pipeline — processes data in time windows ($1=start, $2=end). Best for time-series data.
  • Sequence pipeline — processes data by ID ranges. Best for tables with auto-increment keys.
  • File list pipeline — processes new files on S3. Best for file-based ingestion.
Demo 3: Real-Time Hot → Iceberg Sync

Build the Hot (heap) + Archive (Iceberg) pattern. Frequent writes go to a heap table, and pg_incremental continuously syncs data to an Iceberg archive. A VIEW unifies both for transparent querying.

Architecture: 
[App]
  ↓ INSERT/UPDATE
[events_hot]  (heap — fast writes, recent data)
  ↓ pg_incremental (every 1 minute)
[event_archive]  (Iceberg — S3, long-term storage)

[events_all]  (VIEW = events_hot UNION ALL event_archive_past)
  ↑ PostgreSQL reads via this VIEW

[Snowflake] → reads event_archive directly (Iceberg on S3)
Step 1: Create heap + Iceberg tables + VIEW

The heap table receives live writes. The Iceberg table stores the archive on S3. The VIEW combines them for transparent reads.

-- Hot table (heap — fast writes)
CREATE TABLE IF NOT EXISTS demo.events_hot_live (
    event_id    BIGSERIAL,
    event_date  TIMESTAMPTZ NOT NULL DEFAULT now(),
    region      TEXT NOT NULL,
    category    TEXT NOT NULL,
    amount      NUMERIC(10,2) NOT NULL,
    status      TEXT NOT NULL DEFAULT 'active'
);

-- Archive table (Iceberg — S3 storage)
CREATE TABLE IF NOT EXISTS demo.event_archive (
    event_id    BIGINT,
    event_date  TIMESTAMPTZ NOT NULL,
    region      TEXT NOT NULL,
    category    TEXT NOT NULL,
    amount      NUMERIC(10,2) NOT NULL,
    status      TEXT NOT NULL
) USING iceberg;

-- VIEW: archive data older than today (avoids overlap with hot)
CREATE OR REPLACE VIEW demo.event_archive_past AS
SELECT * FROM demo.event_archive
WHERE event_date < CURRENT_DATE;

-- Unified VIEW: hot + past archive
CREATE OR REPLACE VIEW demo.events_all AS
SELECT event_id, event_date, region, category, amount, status FROM demo.events_hot_live
UNION ALL
SELECT * FROM demo.event_archive_past;
Step 2: Insert sample data into hot table

Simulate 60 days of events in the heap table. In production, this data comes from the application.

INSERT INTO demo.events_hot_live (event_date, region, category, amount, status)
SELECT

Step 3: Initial bulk sync (hot → archive)

Copy all existing data from hot to the Iceberg archive in one operation. This is a one-time bulk load — incremental sync handles new data from here on.

3a: Check archive before sync (should be 0 rows):

SELECT count(*) AS archive_rows FROM demo.event_archive;

3b: Bulk copy all rows:

INSERT INTO demo.event_archive (event_id, event_date, region, category, amount, status)
SELECT event_id, event_date, region, category, amount, status
FROM demo.events_hot_live;

3c: Verify both tables have the same count:

SELECT 'hot' AS source, count(*) AS rows FROM demo.events_hot_live
UNION ALL
SELECT 'archive', count(*) FROM demo.event_archive;
Step 4: Set up incremental sync (pg_incremental + pg_cron)

Now create a pipeline that syncs new data going forward. start_time = now() because historical data is already synced. time_interval = '1 minute' means the pipeline processes data in 1-minute windows. $1 and $2 are the window boundaries — pg_incremental advances these automatically.

4a: Create the pipeline:

CREATE EXTENSION IF NOT EXISTS pg_incremental CASCADE;

SELECT incremental.create_time_interval_pipeline(
    pipeline_name  := 'events_to_archive',
    time_interval  := '1 minute'::interval,
    command        := $$
        INSERT INTO demo.event_archive (event_id, event_date, region, category, amount, status)
        SELECT event_id, event_date, region, category, amount, status
        FROM demo.events_hot_live
        WHERE event_date >= $1 AND event_date < $2
    $$,
    schedule       := NULL,
    start_time     := now()::timestamptz
);

4b: Enable automatic sync with pg_cron (runs every 1 minute):

How this works: pg_cron is installed in the postgres database (a PostgreSQL requirement). It schedules a job that connects to the benchmark database every minute and runs CALL incremental.execute_pipeline().
Step 5: Query via unified VIEW

The events_all VIEW combines hot (heap) and archive (Iceberg). Applications query this single VIEW.

-- Weekly summary across hot + archive
SELECT date_trunc('week', event_date) AS week,
       count(*) AS events,
       sum(amount)::numeric(12,2) AS total_amount
FROM demo.events_all
GROUP BY 1 ORDER BY 1;

This is the core of the demo. INSERT into the hot table, wait ~1 minute, and confirm the data appears in the Iceberg archive automatically.

6a: INSERT distinctive rows into hot:

Wait ~1 minute for pg_cron to sync, then check the archive:

6b: Check archive (should appear after ~1 minute):

FROM demo.event_archive WHERE category = 'demo_live' ORDER BY event_date;

DELETE FROM demo.events_hot_live WHERE category = 'demo_live';
Note: pg_incremental syncs INSERTs only (append-only pipeline). UPDATEs and DELETEs on the hot table are not propagated to the archive. If historical data needs correction, UPDATE the archive directly (Step 7).
Step 7: Rare update on archive (exception operation)

Normally, only events_hot_live receives writes. But if historical data needs correction, you can UPDATE the Iceberg archive directly:

-- Correct a past event directly on the archive
UPDATE demo.event_archive
SET status = 'corrected', amount = 0
WHERE event_date < now() - interval '30 days'
  AND category = 'refund'
  AND region = 'Tokyo'
  AND event_id IN (
      SELECT event_id FROM demo.event_archive
      WHERE event_date < now() - interval '30 days'
        AND category = 'refund' AND region = 'Tokyo'
      LIMIT 5
  );

-- Verify
SELECT event_id, event_date, status, amount
FROM demo.event_archive
WHERE status = 'corrected'
ORDER BY event_date LIMIT 5;
Note: Iceberg UPDATEs are heavier than heap UPDATEs (they rewrite Parquet files on S3). Keep archive direct updates to exceptional corrections only.
