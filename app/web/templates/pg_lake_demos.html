What Setup does:
  1. CREATE TABLE ... USING iceberg — An Iceberg table looks like a normal PostgreSQL table, but data is stored as Parquet files on S3 instead of local disk.
  2. WITH (partition_by = 'day(log_time)') — pg_lake automatically organizes Parquet files by day on S3, enabling efficient time-range queries.
  3. INSERT 30,000 sample rows (60 days of access logs) — data flows through PostgreSQL and lands on S3 as Parquet.
Check S3: After Setup, open your S3 bucket in the AWS Console. You will see a directory structure like s3://bucket/.../tables/benchmark/lake_demo/access_logs/.../data/ with UUID-named folders containing Parquet files. This confirms that PostgreSQL INSERT actually wrote data to S3.
Step 1: Aggregate Query with Partition Pruning

Query the Iceberg table. pg_lake pushes the query to DuckDB, which reads Parquet from S3 and filters by the date range.

SELECT date_trunc('day', log_time) AS day,
       count(*) AS requests,
       avg(response_ms)::numeric(10,2) AS avg_ms
FROM lake_demo.access_logs
WHERE log_time >= now() - interval '7 days'
GROUP BY 1 ORDER BY 1;
Reading the EXPLAIN output:
  • Custom Scan (Query Pushdown) / Engine: DuckDB — PostgreSQL delegated execution to DuckDB (pg_lake's internal columnar engine)
  • READ_PARQUET with Filters: — DuckDB reads Parquet from S3 and applies the WHERE clause pushdown
  • This is fundamentally different from a normal PostgreSQL Seq Scan — the data never existed on local disk
Step 2: Single-Row INSERT (verify DML works)

Iceberg tables support standard INSERT/UPDATE/DELETE, just like regular PostgreSQL tables.

INSERT INTO lake_demo.access_logs (log_time, user_id, action, path, status, response_ms)

What is a Foreign Table?
  • A "virtual table" that references data on S3 without copying it into PostgreSQL.
  • pg_lake uses DuckDB internally to scan Parquet files in columnar format — fast aggregation, filter pushdown.
  • Uses PostgreSQL's Foreign Data Wrapper (FDW) mechanism — standard SQL, standard permissions.
  • Prerequisite: Run Demo 2 first to create the Parquet files on S3.

Why partitioning matters: As tables grow (millions/billions of rows), full table scans become slow. Partitioning splits the table into smaller physical pieces. When a query includes a WHERE clause on the partition key, PostgreSQL skips partitions that cannot contain matching rows — this is called partition pruning. Demo 4 builds on this: old partitions are moved to S3 via FDW, while the query interface stays the same.

ATTACH / DETACH — The Core of Hot/Cold Architecture

PostgreSQL allows you to detach a partition (making it a standalone table) and attach any table — including a Foreign Table — as a partition. This is the key technique used in Demo 4.

OperationWhat it doesUse case
DETACH PARTITION Removes a partition from the parent table. Data is preserved as a standalone table. Prepare a partition for archival or storage migration
ATTACH PARTITION Adds an existing table (heap or Foreign Table) as a partition of the parent. Re-attach after migration, or attach an S3 FDW as a cold partition

Since PostgreSQL 11, Foreign Tables can be attached as partitions. This enables the hybrid heap + S3 architecture in Demo 4.

{{ t('lake.demo4.title') }}

{{ t('lake.demo4.desc') }}

The story: IoT sensors send data every second. The last 3 days are kept on fast local storage (heap) for real-time queries and updates. Data older than 3 days is automatically archived to S3 in Iceberg format. Applications query a single table name — PostgreSQL handles the routing transparently. Snowflake can also read the same Iceberg data on S3 (zero ETL).
{{ t('lake.demo4.arch_note') }}
{{ t('lake.demo4.arch_diagram') }}
Step 1: {{ t('lake.demo4.step1') }}

Install the required extensions, create a partitioned table, and let pg_partman manage daily partitions automatically. create_parent() creates today's partition plus 3 days ahead (p_premake), and a default partition for data outside any range. 

-- Install required extensions
CREATE EXTENSION IF NOT EXISTS pg_partman CASCADE;
CREATE EXTENSION IF NOT EXISTS pg_incremental CASCADE;

CREATE SCHEMA IF NOT EXISTS iot;

CREATE TABLE iot.sensor_data (
    ts        TIMESTAMPTZ NOT NULL,
    device_id INT         NOT NULL,
    temp      DOUBLE PRECISION,
    humidity  DOUBLE PRECISION,
    pressure  DOUBLE PRECISION
) PARTITION BY RANGE (ts);

SELECT create_parent(
    p_parent_table := 'iot.sensor_data',
    p_control      := 'ts',
    p_interval     := '1 day',
    p_premake      := 3
);

After running, verify the partitions that pg_partman created:

SELECT inhrelid::regclass AS partition, pg_get_expr(relpartbound, inhrelid) AS bounds
FROM pg_inherits JOIN pg_class ON pg_class.oid = inhrelid
WHERE inhparent = 'iot.sensor_data'::regclass ORDER BY partition;
Step 2: {{ t('lake.demo4.step2') }}

Create a separate Iceberg table as the sync target. USING iceberg means data lives on S3 as Parquet with Iceberg metadata. pg_incremental will copy data from the heap partitions into this table. 

CREATE TABLE iot.sensor_archive (
    ts        TIMESTAMPTZ NOT NULL,
    device_id INT         NOT NULL,
    temp      DOUBLE PRECISION,
    humidity  DOUBLE PRECISION,
    pressure  DOUBLE PRECISION
) USING iceberg
  WITH (partition_by = 'day(ts)');
Step 3: {{ t('lake.demo4.step3_pipe') }}

Set up a pg_incremental pipeline that syncs data from heap to Iceberg. $1 and $2 are the time window boundaries (start/end) — pg_incremental tracks progress automatically. schedule := NULL because pg_cron is not available in this database; we'll run it manually. start_time is set to 60 days ago to cover all sample data. 

SELECT incremental.create_time_interval_pipeline(
    pipeline_name  := 'iot_to_iceberg',
    time_interval  := '1 day'::interval,
    command        := $$
        INSERT INTO iot.sensor_archive (ts, device_id, temp, humidity, pressure)
        SELECT ts, device_id, temp, humidity, pressure
        FROM iot.sensor_data
        WHERE ts >= $1 AND ts < $2
    $$,
    schedule       := NULL,
    start_time     := (now() - interval '60 days')::timestamptz
);
Step 4: {{ t('lake.demo4.step4_data') }}

Insert 300,000 rows spanning 60 days. pg_partman automatically routes each row to the correct daily partition based on ts.

INSERT INTO iot.sensor_data (ts, device_id, temp, humidity, pressure)
SELECT
    now() - (random() * interval '60 days'),
    (random() * 100)::int,
    20.0 + random() * 15.0,
    40.0 + random() * 40.0,
    1000.0 + (random() - 0.5) * 50.0
FROM generate_series(1, 300000) AS s(i);
Step 5: {{ t('lake.demo4.step5_sync') }}

Execute the pipeline manually. Each CALL processes one day's worth of data. Since we have 60 days and time_interval = '1 day', you need to call it ~60 times. In production, pg_cron runs this automatically every minute.

5a: Check Iceberg archive before sync (should be 0 rows):

SELECT count(*) AS archive_rows FROM iot.sensor_archive;

5b: Run the pipeline. With batched = true (default), a single call processes all pending days from start_time to now:

CALL incremental.execute_pipeline('iot_to_iceberg');

5c: Verify sync — check archive row count and daily distribution:

SELECT date_trunc('day', ts) AS day, count(*) AS rows
FROM iot.sensor_archive
GROUP BY 1 ORDER BY 1;
Step 6: {{ t('lake.demo4.step6_switch') }}

{{ t('lake.demo4.step6_desc') }}

What happens in this step (4 operations):
  1. DETACH — Remove old heap partitions from the parent table. Data is preserved as standalone tables.
  2. DROP — Delete the heap data (it's already synced to S3 Iceberg in Step 5).
  3. CREATE FOREIGN TABLE — Create a virtual table pointing to the Iceberg Parquet files on S3.
  4. ATTACH — Re-attach the Foreign Table as a partition. Now queries on the parent table transparently read S3 for old data.
-- will be generated dynamically based on S3 prefix

Verify: partitions now show a mix of heap and FOREIGN TABLE:

SELECT inhrelid::regclass AS partition,
       pg_get_expr(relpartbound, inhrelid) AS bounds,
       CASE WHEN c.relkind = 'f' THEN 'FOREIGN TABLE (S3)' ELSE 'heap' END AS storage
FROM pg_inherits JOIN pg_class c ON c.oid = inhrelid
WHERE inhparent = 'iot.sensor_data'::regclass ORDER BY partition;
Step 7: {{ t('lake.demo4.step7_verify') }}

Run queries with different time ranges to see partition pruning in action. Narrow ranges skip Cold (S3) partitions entirely.

A: Yesterday only — heap only, zero S3 access:

EXPLAIN (COSTS OFF)
SELECT avg(temp), avg(humidity) FROM iot.sensor_data
WHERE ts >= now() - interval '1 day';

B: This week — heap + some S3 partitions:

SELECT date_trunc('day', ts) AS day, count(*), avg(temp)::numeric(5,2) AS avg_temp
FROM iot.sensor_data WHERE ts >= now() - interval '7 days'
GROUP BY 1 ORDER BY 1;

C: This month — heap + ~27 Cold partitions:

SELECT date_trunc('day', ts) AS day, count(*), avg(temp)::numeric(5,2) AS avg_temp
FROM iot.sensor_data WHERE ts >= date_trunc('month', now())
GROUP BY 1 ORDER BY 1;

D: Full range — all partitions (heap + FDW). One table name, 60 days of data across two storage backends:

SELECT date_trunc('week', ts) AS week, count(*), avg(temp)::numeric(5,2) AS avg_temp
FROM iot.sensor_data
GROUP BY 1 ORDER BY 1;
{{ t('lake.sf_ref.title') }}

{{ t('lake.sf_ref.desc') }}

Why this matters: The Iceberg data that pg_lake wrote to S3 (Demo 1, Demo 4) can be read directly by Snowflake — no ETL pipeline needed. Snowflake registers the Iceberg metadata and reads the same Parquet files. Run ALTER ICEBERG TABLE ... REFRESH to pick up new data.

Run the following SQL in a Snowflake Worksheet (not in this app):

-- === Snowflake: Read Iceberg table from S3 ===

-- 1. External Volume (S3 bucket access via IAM role)
CREATE OR REPLACE EXTERNAL VOLUME pg_lake_vol
    STORAGE_LOCATIONS = ((
        NAME = 'pg_lake_s3',
        STORAGE_BASE_URL = 's3://your-bucket/pg_lake/',
        STORAGE_PROVIDER = 'S3',
        STORAGE_AWS_ROLE_ARN = 'arn:aws:iam::123456789012:role/pg-lake-access'
    ));

-- 2. Catalog Integration (tells Snowflake how to read Iceberg metadata)
CREATE OR REPLACE CATALOG INTEGRATION pg_lake_catalog
    CATALOG_SOURCE = OBJECT_STORE
    TABLE_FORMAT = ICEBERG
    ENABLED = TRUE;

-- 3. Register Iceberg Table (no data copy — reads S3 directly)
CREATE OR REPLACE ICEBERG TABLE analytics.sensor_from_pg
    EXTERNAL_VOLUME = 'pg_lake_vol'
    CATALOG = 'pg_lake_catalog'
    METADATA_FILE_PATH = 'sensor_archive/metadata/v1.metadata.json';

-- 4. Query (same data that pg_lake wrote from PostgreSQL)
SELECT date_trunc('hour', ts) AS hour,
       avg(temp) AS avg_temp, count(*) AS readings
FROM analytics.sensor_from_pg
GROUP BY 1 ORDER BY 1;

-- 5. Refresh metadata (after new data is written by pg_lake)
ALTER ICEBERG TABLE analytics.sensor_from_pg REFRESH;
